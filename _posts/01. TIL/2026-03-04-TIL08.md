## 2. C#

- const와 readonly

const는 컴파일타임에 결정되는 상수. const는 컴파일타임에 `리터럴`로 컴파일러가 교체시키기 때문에 런타임(생성자)에서 기입할 수 없음. 클래스에서 const는 static 키워드처럼 취급됨. 즉, 클래스에 const를 선언하면 모든 인스턴스가 같은 값을 공유함. CPP의 constexpr과 유사. const는 변수가 할당되지 않아서 메모리에 잡히지 않음. 그래서 int, bool, enum, string등만 가능함. 그래서 `[SerializeField]를 사용할 수 없음.

readonly는 런타임에 결정되는 상수. readonly는 인스턴스마다 다른 값을 가질 수 있음. CPP의 const를 사용하는 의미가 readonly와 유사.

둘을 사용할 때 dll로 만들 때 차이가 존재. const는 값이 변경되면 컴파일을 다시해야하지만, readonly는 변수의 주소를 참조하고 있는 것이기 때문에 컴파일을 다시 하지 않아도 된다고 함.

- object와 valueType

모든 타입은 object에서 파생됨. ValueType은 object를 상속받음. 그러나, 개념적 상속 구조를 갖고는 있으나, 물리적으로 상속은 아님. 

int, struct, enum등은 ValueType을 상속받음. ValueType은 값 타입이며, 스택에 저장됨. 리턴이나 파라미터로 전달하는 경우 깊은 복사가 진행됨.

object는 참조 타입이며, 힙에 저장됨. 리턴이나 파라미터로 전달하는 경우 얕은 복사가 진행됨. object는 가비지 컬렉터에 의해 관리됨. 추가적으로 object는 타입 정보와 동기화 블록 인덱스가 존재하는게 차이점.

- boxing과 unboxing

boxing이란 값 타입을 참조 타입으로 변경하는 과정. unboxing은 그 반대.

boxing을 하게되면 스택에 있던 메모리를 힙에 복사(GC 추적)하여 저장하게 되며, 타입 정보와 동기화 블록을 만드는 과정을 거침. object 타입이나 ICompareable타입으로 변경하게 될 때 boxing이 발생함. 범용적인 처리를 위해 object 타입으로 파라미터를 받는 API라던가, List가 아니라 ArrayList를 사용하게 되면 boxing이 발생함. object에 들어있는 메서드(`ToString()`, `GetType()`)등을 사용하게 되면 boxing이 발생함.

- ref와 out

ref는 이미 사용하던 객체를 함수 내에서 수정할 수 있도록 만드는 것. out은 함수 내에서 값을 할당하고 반환받게 되는 것을 명시함. 일반 참조 타입은 전달하면 어차피 수정되지 않는가? 그럼 ref를 쓰는 이유가 뭐지? => ref로 넘기게 되면 참조하는 공간 자체를 변경할 수 있음

- for와 foreach

foreach는 for보다 비쌈. 왜냐하면, IEnumerator를 생성해야 하기도 하고, MoveNext등의 함수를 호출하는 등 별도의 작업들이 존재하기 때문에 비쌈. 그러나 IEnumerator가 존재하는 컨테이너에 대해 범위는 정확하게 동작.

- using(GC의 managed, unmanaged)

GC에게 관리되는 객체와 관리되지 않는 객체가 존재함. 이 객체들이 정상적으로 `Dispose()`할 수 있도록 함

- 코루틴의 상태머신

IEnumerator MyCoroutine() {
    int x = 10;          // (A)
    yield return null;   // (B) 멈춤 지점 1
    x++;                 // (C)
    yield return null;   // (D) 멈춤 지점 2
}

bool MoveNext() {
    switch (this.state) {
        case 0: // 처음 실행 시
            this.x = 10; // (A) 실행
            this.state = 1; // 다음 위치 저장
            return true; // (B) 여기서 일단 중단! 리턴함.
        
        case 1: // 다시 불렸을 때
            this.x++; // (C) 아까 저장된 x를 가져와서 계산
            this.state = 2; // 다음 위치 저장
            return true; // (D) 여기서 다시 중단!
            
        default:
            return false; // 더 이상 실행할 코드가 없음
    }
}

코루틴은 내부적으로 상태 머신으로 동작한다고 함

- 탈 가상화

탈 가상화란? 가상화가 됐던 클래스 또는 메서드를 파생클래스에서 가상화가 되지 않도록 만드는 것.

왜 하는가? 가상화가 되면 가상함수 테이블을 거쳐서 함수 호출을 해야하기 때문에, 간접 참조 및 캐시 히트율 때문에 가상화 되지 않은 함수가 성능적으로 좋음.

C#에선 Sealed 라는 키워드를 붙이면 됨. cpp에서 final도 같은 기능을 함
