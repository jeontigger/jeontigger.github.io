현재 빌드&컴파일 폴더에는 5개의 파일이 있습니다:

- `02.컴파일_어셈블_링크.md` - 컴파일/어셈블/링크 단계 ⚠️ (GPT 답변 누락)
- `03.헤더가드_prgmaonce.md` - 헤더 가드 vs #pragma once ✅

- `05.ODR.md` - ODR(One Definition Rule) ⚠️ (GPT 답변 누락)

**전체 상태:** ⚠️ 일부 파일 GPT 답변 누락

### 1.2 파일별 상세 분석

#### `01.전처리.md` - ✅ 양호

**강점:**
- GPT 답변이 완성되어 있음
- 기본 개념 → 주요 역할 → 조건부 컴파일 → 매크로 관련 주의점 → 헤더 가드 → 전처리 결과 → 디버깅/설계 판단 순서로 잘 구성됨
- 질문의 깊이가 적절함

**개선 필요 사항:**
1. **사용자 답변 공간 없음**
   - 일관성을 위해 추가 고려 (선택사항)

2. **파일명 오타**
   - `03.헤더가드_prgmaonce.md` → `prgmaonce`는 `pragmaonce`의 오타로 보임
   - 실제로는 `pragmaonce`가 맞지만, 파일명에는 언더스코어가 없음

#### `02.컴파일_어셈블_링크.md` - ⚠️ GPT 답변 완전히 누락

**강점:**
- 질문 구조가 잘 되어 있음: 전체 흐름 → 컴파일 단계 → 어셈블 단계 → 링크 단계 → 오브젝트 파일/심볼 → 정적/동적 링크
- 카테고리별로 질문이 체계적으로 구성됨

**문제점:**
1. **GPT 답변 섹션이 비어있음**
   - GPT 답변 섹션에 질문만 반복되어 있고 실제 답변이 없음
   - 모든 질문에 대한 답변 필요

2. **사용자 답변 공간 없음**
   - 다른 파일과 달리 사용자 답변 공간이 제공되지 않음
   - 일관성을 위해 추가 고려

#### `03.헤더가드_prgmaonce.md` - ✅ 양호

**강점:**
- GPT 답변이 완성되어 있음
- 기본 개념 → 동작 방식 → 표준/이식성 → 성능/빌드 관점 → 안정성/문제점 → 실무 선택 기준 → 요약 순서로 잘 구성됨
- 헤더 가드와 #pragma once의 차이점이 명확함

**개선 필요 사항:**
1. **사용자 답변 공간 없음**
   - 일관성을 위해 추가 고려 (선택사항)

2. **파일명 오타 확인 필요**
   - `prgmaonce`가 의도된 것인지 확인 필요

#### `04.정적라이브러리vs동적.md` - ✅ 양호

**강점:**
- GPT 답변이 완성되어 있음
- 기본 개념 → 링크 시점 차이 → 실행 파일 관점 → 메모리 사용 → 배포/유지보수 → 성능 관점 → 사용 판단 → 요약 순서로 잘 구성됨
- 정적/동적 라이브러리의 차이점이 명확함

**개선 필요 사항:**
1. **사용자 답변 공간 없음**
   - 일관성을 위해 추가 고려 (선택사항)

#### `05.ODR.md` - ⚠️ GPT 답변 완전히 누락

**강점:**
- 질문 구조가 잘 되어 있음: 기본 개념 → 선언 vs 정의 → 위반 사례 (핵심) → inline/template/constexpr → 링크 단계 관점 → 설계/예방 → 요약
- 카테고리별로 질문이 체계적으로 구성됨

**문제점:**
1. **GPT 답변 섹션이 비어있음**
   - GPT 답변 섹션에 질문만 반복되어 있고 실제 답변이 없음
   - 모든 질문에 대한 답변 필요

2. **사용자 답변 공간 없음**
   - 다른 파일과 달리 사용자 답변 공간이 제공되지 않음
   - 일관성을 위해 추가 고려

### 1.3 전체적인 문제점 요약

1. **일관성 부족**
   - 모든 파일에 사용자 답변 공간이 없음
   - GPT 답변 형식은 대체로 일관되지만, 2개 파일은 완전히 비어있음

2. **누락된 주제**
   - CMake / 빌드 시스템
   - 컴파일러 최적화 옵션 (-O, -O2, -O3)
   - 링커 스크립트
   - 심볼 테이블과 링킹 과정 상세

3. **질문의 깊이 불균형**
   - 대부분의 파일: GPT 답변이 완성되어 있음
   - `02.컴파일_어셈블_링크.md`, `05.ODR.md`: GPT 답변 섹션이 완전히 비어있음

## 4. 질문 형식 표준화 (standardize_format)

### 4.1 권장 파일 구조

```markdown
# 🧠 [주제] 면접 질문 리스트

## ✅ [카테고리 1]

1. [질문]
   
   - [사용자 답변 공간] (선택사항)

2. [질문]
   
   - [사용자 답변 공간] (선택사항)

## ✅ [카테고리 2]

...

<details>
<summary>GPT 1줄 답변</summary>
<div markdown="1">

## 📝 모범 답변

### [카테고리 1]

1. [답변]
2. [답변]

### [카테고리 2]

...

</div>
</details>

---

## GPT 피드백

[필요시 피드백 추가]
```

### 4.2 질문 작성 가이드라인

#### 질문 수준
- **기본 개념**: 면접 초반에 나올 수 있는 기본적인 질문
- **심화 개념**: 내부 구조, 동작 원리, 각 단계의 상세 과정
- **실무 적용**: 언제 사용하는지, 어떤 경우에 선택하는지

#### 질문 순서
1. 기본 개념 질문 (2-3개)
2. 내부 구조/동작 원리 질문 (3-5개)
3. 다른 방식과의 비교 질문 (2-4개)
4. 실무 적용/선택 기준 질문 (1-2개)
5. 요약 질문 (1개)

#### 필수 포함 질문 유형
- [ ] 기본 개념 질문 포함
- [ ] 내부 구조/동작 원리 질문 포함
- [ ] 다른 방식과의 비교 질문 포함 (해당되는 경우)
- [ ] 실무 적용/선택 기준 질문 포함
- [ ] 요약 질문 포함

### 4.3 답변 작성 가이드라인

#### GPT 답변 형식
- **1줄 답변**: 간결하고 핵심만 담은 답변
- **카테고리별 구분**: 질문 카테고리와 동일하게 구성
- **번호 매칭**: 질문 번호와 답변 번호 일치

#### 답변 내용 기준
- **정확성**: 기술적으로 정확한 정보 제공
- **간결성**: 면접에서 답변할 수 있을 정도로 간결
- **완전성**: 질문의 핵심을 모두 다룸
- **예시 포함**: 필요시 코드 예시나 구체적 예시 포함

#### 사용자 답변 공간
- **선택사항**: 모든 질문에 필수는 아님
- **일관성**: 파일 내에서 일관되게 적용
- **형식**: 4칸 들여쓰기로 답변 공간 제공

### 4.4 카테고리 명명 규칙

**표준 카테고리:**
- `## ✅ 기본 개념`
- `## ✅ [단계명] 단계` (전처리, 컴파일, 어셈블, 링크 등)
- `## ✅ [다른 방식]과의 비교` (해당되는 경우)
- `## ✅ 실무 / 설계 판단`
- `## ✅ 요약 질문`

**주제별 추가 카테고리:**
- 빌드 과정 관련: `## ✅ 전체 흐름`, `## ✅ 입력과 출력`
- 링킹 관련: `## ✅ 링크 시점 차이`, `## ✅ 메모리 사용`
- 헤더 관리 관련: `## ✅ 동작 방식`, `## ✅ 표준 / 이식성`

## 7. 참고 사항

### 파일명 규칙
- 형식: `##.주제명.md` (##는 2자리 숫자)
- 예시: `01.전처리.md`, `06.컴파일러_최적화.md`

### 마크다운 형식
- 제목: `# 🧠 [주제] 면접 질문 리스트`
- 카테고리: `## ✅ [카테고리명]`
- 질문: 번호와 함께 작성
- 사용자 답변: 4칸 들여쓰기
- GPT 답변: `<details>` 태그 내부에 작성

### 질문 작성 시 주의사항
- 면접에서 실제로 나올 수 있는 수준으로 작성
- 기본 개념부터 심화 개념까지 단계적으로 구성
- 빌드 과정 질문은 각 단계의 입력/출력을 명확히
- 링킹 질문은 심볼 해석과 메모리 레이아웃 고려
