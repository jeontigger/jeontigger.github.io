## ✅ 기본 개념

1. 소켓(Socket)이란 무엇인가요?

    - 원격 노드와 소통하기 위한 창구입니다.

2. 소켓이 네트워크 프로그래밍에서 어떤 역할을 하나요?

    - 데이터를 송수신하는 역할입니다.

3. 소켓과 파일 디스크립터의 관계는 무엇인가요?

    - 소켓은 파일 디스크립터와 같습니다.

## ✅ 소켓 종류

1. TCP 소켓과 UDP 소켓의 차이는 무엇인가요?

    - TCP 소켓은 다른 TCP 소켓과 연결되어있어야 하고, UDP 소켓은 열려있기만 해도 데이터를 받을 수 있습니다.

2. SOCK_STREAM과 SOCK_DGRAM의 차이는 무엇인가요?

    - SOCK_STREAM은 TCP용 옵션, SOCK_DGRAM은 UDP용 옵션입니다.

3. 언제 TCP 소켓을 사용하고, 언제 UDP 소켓을 사용하나요?

    - 연결해서 신뢰성이 필요한 패킷인 경우에 TCP 소켓을 사용하고, 빠른 데이터 송수신이 필요할 경우에 UDP 소켓을 사용하는 것이 좋습니다.

## ✅ 네트워크 바이트 순서

1. 네트워크 바이트 순서(Network Byte Order)란 무엇인가요?

    - 네트워크에서 바이트 스트림을 읽는 순서를 의미합니다.

2. 왜 바이트 순서 변환이 필요한가요?

    - 장치에서는 주로 리틀인디언으로 사용하고, 네트워크는 주로 빅 인디언으로 바이트를 읽기 때문에 변환이 필요합니다.

3. htonl, htons, ntohl, ntohs 함수의 역할은 무엇인가요?

    - h는 host, n은 network를 뜻합니다. host는 리틀 인디언, network는 빅 인디언으로 변환합니다. l은 64비트, s는 32비트를 뜻합니다.

4. 빅엔디안(Big-Endian)과 리틀엔디안(Little-Endian)의 차이는 무엇인가요?

    - 빅 인디언은 사람이 읽기 편한 방식으로 바이트를 리딩하고, 리틀 인디언은 컴퓨터가 읽기 편한식의 리딩 방식입니다.

### ✅ 기본 개념

1. 네트워크 통신을 위한 엔드포인트(종단점)로, 프로세스 간 통신을 가능하게 하는 추상화된 인터페이스입니다.
2. 네트워크 통신의 추상화 계층으로, 애플리케이션이 네트워크 프로토콜의 복잡성을 숨기고 통신할 수 있게 합니다.
3. 소켓은 파일 디스크립터로 표현되며, 파일처럼 read/write 연산을 수행할 수 있습니다.

### ✅ 소켓 종류

1. TCP 소켓은 연결 지향적이고 신뢰성이 있으며, UDP 소켓은 비연결형이고 빠르지만 신뢰성이 없습니다.
2. SOCK_STREAM은 TCP(연결 지향), SOCK_DGRAM은 UDP(데이터그램)를 의미합니다.
3. 신뢰성이 필요한 경우 TCP, 속도가 중요하고 일부 손실이 허용되는 경우 UDP를 사용합니다.

### ✅ 소켓 주소 구조체

1. 다양한 주소 체계(IPv4, IPv6, Unix 등)를 하나의 인터페이스로 통일하기 위해서입니다.
2. sockaddr_in은 IPv4, sockaddr_in6은 IPv6 주소를 저장합니다.
3. 주소 체계를 나타내며, AF_INET(IPv4), AF_INET6(IPv6) 등의 값을 가집니다.
4. inet_addr() 또는 inet_aton() 함수를 사용하거나, 직접 sin_addr.s_addr에 네트워크 바이트 순서로 변환된 값을 설정합니다.

### ✅ 네트워크 바이트 순서

1. 네트워크에서 사용하는 표준 바이트 순서로, 빅엔디안을 사용합니다.
2. 호스트마다 바이트 순서가 다르기 때문에, 네트워크 전송 시 통일된 형식이 필요합니다.
3. htonl/htons는 호스트에서 네트워크로, ntohl/ntohs는 네트워크에서 호스트로 변환합니다.
4. 빅엔디안은 상위 바이트부터, 리틀엔디안은 하위 바이트부터 저장하는 방식입니다.

### ✅ 소켓 도메인

1. AF_INET과 PF_INET은 거의 동일하며, AF는 주소 체계, PF는 프로토콜 패밀리를 의미합니다.
2. AF_INET은 IPv4, AF_INET6은 IPv6를 의미합니다.
3. 같은 머신 내 프로세스 간 통신에 사용하며, 네트워크 스택을 거치지 않아 빠릅니다.

### ✅ 실무 / 설계 판단

1. IPv6가 미래 지향적이지만, 현재는 IPv4가 더 널리 사용되므로 호환성을 고려해야 합니다.
2. 지원할 주소 체계, 크로스 플랫폼 호환성, 메모리 사용량 등을 고려해야 합니다.

    - host가 항상 리틀엔디안은 아닙니다 (빅엔디안 시스템도 존재)
    - l은 32비트(long), s는 16비트(short)입니다 (64비트가 아님)
  - ✅ **정확한 답변**: "htonl/htons는 호스트에서 네트워크로, ntohl/ntohs는 네트워크에서 호스트로 변환합니다"
  - htonl = host to network long (32비트), htons = host to network short (16비트)
  - 변환 방향(host↔network)과 데이터 크기(long/short)를 정확히 이해해야 합니다

**질문 4: 빅엔디안(Big-Endian)과 리틀엔디안(Little-Endian)의 차이는 무엇인가요?**
- **현재 답변**: "빅 인디언은 사람이 읽기 편한 방식으로 바이트를 리딩하고, 리틀 인디언은 컴퓨터가 읽기 편한식의 리딩 방식입니다"
- **피드백**:
  - ⚠️ **부정확**: "사람이 읽기 편한", "컴퓨터가 읽기 편한"이라는 표현은 부정확합니다
  - ✅ **정확한 답변**: "빅엔디안은 상위 바이트부터, 리틀엔디안은 하위 바이트부터 저장하는 방식입니다"
  - 바이트 저장 순서의 차이이며, 각각의 장단점이 있지만 "편한" 정도의 문제는 아닙니다

### 📊 전체 평가

**강점:**
- ✅ 소켓의 기본적인 역할(데이터 송수신)을 이해하고 있음
- ✅ TCP/UDP의 기본적인 차이(연결 여부)를 알고 있음
- ✅ SOCK_STREAM과 SOCK_DGRAM이 각각 TCP/UDP용임을 알고 있음
- ✅ IPv4/IPv6의 기본적인 구분을 알고 있음

**개선 필요:**
- ❌ **오개념**: 
  - htonl/htons 함수의 비트 크기 (l=32비트, s=16비트, 64비트 아님)
  - host가 항상 리틀엔디안이라는 가정 (빅엔디안 시스템도 존재)
- ⚠️ **부정확한 부분**:
  - 소켓과 파일 디스크립터의 관계 ("같다" vs "표현된다")
  - 네트워크 바이트 순서의 정의 ("읽는 순서" vs "표준 바이트 순서")
  - sockaddr의 목적 ("설정 정보" vs "주소 체계 통일")
  - 빅엔디안/리틀엔디안의 차이 설명
- 💡 **개선이 필요한 부분**:
  - 소켓 정의에 "엔드포인트", "추상화" 용어 추가
  - 소켓의 역할에 "추상화 계층" 개념 추가
  - TCP/UDP 차이에 "신뢰성" 개념 추가
  - UDP 사용 조건에 "일부 손실 허용" 추가
  - IPv4/IPv6 선택 기준을 더 구체적으로 설명
- ❌ **답변 누락**: 
  - sockaddr의 sa_family 필드
  - IPv4 주소 설정 방법
  - AF_INET과 PF_INET의 차이
  - AF_INET과 AF_INET6의 차이
  - AF_UNIX 소켓 사용 시기
  - 소켓 주소 구조체 선택 고려사항
  - 소켓 기본 개념 요약

**우선순위:**
1. **높음**: 
   - 네트워크 바이트 순서 변환 함수의 정확한 이해 (htonl/htons의 비트 크기, 변환 방향)
   - 답변 누락된 기본 개념들 (sa_family, 주소 설정 방법, AF_INET/PF_INET 차이)
2. **중간**: 
   - 소켓과 파일 디스크립터의 정확한 관계
   - TCP/UDP 차이에 신뢰성 개념 추가
   - sockaddr의 목적 정확히 이해
3. **낮음**: 
   - 용어의 정확성 개선 (엔드포인트, 추상화 등)
   - IPv4/IPv6 선택 기준 구체화

**특별 주의사항:**
- 네트워크 바이트 순서 변환 함수는 면접에서 자주 나오는 주제입니다. htonl/htons의 정확한 의미와 비트 크기를 반드시 암기해야 합니다
- sockaddr 구조체와 관련 함수들(socket(), bind(), connect() 등)은 소켓 프로그래밍의 핵심이므로 정확히 이해해야 합니다
- TCP/UDP의 차이는 단순히 연결 여부가 아니라, 신뢰성과 속도의 트레이드오프를 이해하는 것이 중요합니다
