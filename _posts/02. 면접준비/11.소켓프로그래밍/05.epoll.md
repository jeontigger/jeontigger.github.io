## 📝 모범 답변

### ✅ 기본 개념

1. Linux에서 제공하는 고성능 I/O 멀티플렉싱 메커니즘입니다.
2. select/poll의 성능 한계를 해결하기 위해 등장했습니다.
3. O(1) 복잡도와 대규모 연결 처리 능력입니다.

### ✅ epoll API

1. epoll 인스턴스를 생성하고 파일 디스크립터를 반환합니다.
2. epoll 인스턴스에 파일 디스크립터를 추가/수정/삭제합니다.
3. 준비된 파일 디스크립터를 대기하고 반환합니다.
4. events(이벤트 플래그), data(사용자 데이터) 필드가 있습니다.

### ✅ 레벨 트리거 vs 엣지 트리거

1. 파일 디스크립터가 준비 상태인 동안 계속 이벤트를 발생시킵니다.
2. 파일 디스크립터 상태가 변경될 때만 이벤트를 발생시킵니다.
3. 레벨 트리거는 계속 알림, 엣지 트리거는 상태 변경 시 한 번만 알림입니다.
4. 일반적으로는 레벨 트리거, 고성능이 필요하면 엣지 트리거를 사용합니다.

### ✅ epoll 사용 패턴

1. epoll_create() → epoll_ctl()로 fd 등록 → epoll_wait()로 대기 → 이벤트 처리 → 반복입니다.
2. ADD는 추가, MOD는 수정, DEL은 삭제를 의미합니다.
3. EPOLLIN은 읽기 가능, EPOLLOUT은 쓰기 가능, EPOLLERR는 에러, EPOLLHUP는 연결 종료입니다.

### ✅ 성능 특성

1. O(1) 복잡도로, 준비된 파일 디스크립터만 반환합니다.
2. 커널이 준비된 파일 디스크립터만 추적하므로 효율적입니다.
3. 수만 개의 연결도 효율적으로 처리할 수 있습니다.

### ✅ select/poll과의 비교

1. epoll은 커널이 준비 상태를 추적하지만, select는 매번 전체를 확인합니다.
2. epoll은 무제한 파일 디스크립터를 지원하지만, poll은 제한이 있습니다.
3. Linux 전용이며, 다른 플랫폼에서는 사용할 수 없습니다.

### ✅ 실무 사용

1. 엣지 트리거 모드에서는 논블로킹 소켓을 사용해야 합니다.
2. 모든 데이터를 읽어야 하며, EAGAIN이 나올 때까지 반복해야 합니다.
3. 엣지 트리거 모드에서 모든 데이터를 처리하기 위해서입니다.

### ✅ 실무 / 설계 판단

1. 대규모 연결을 처리하거나 고성능이 필요한 서버에서 사용합니다.
2. 소규모 연결에서는 select/poll로도 충분합니다.
3. kqueue(BSD/macOS), IOCP(Windows) 등이 있습니다.

### ✅ 요약 질문

1. epoll은 Linux의 고성능 I/O 멀티플렉싱으로, O(1) 복잡도와 대규모 연결 처리 능력을 제공합니다.
