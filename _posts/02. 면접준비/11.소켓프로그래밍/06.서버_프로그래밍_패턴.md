## 📝 모범 답변

### ✅ 기본 개념

1. 한 번에 하나의 클라이언트만 처리하는 서버입니다.
2. 여러 클라이언트를 동시에 처리하는 서버입니다.
3. 반복 서버는 순차 처리, 동시성 서버는 병렬 처리입니다.

### ✅ 반복 서버

1. accept() → 처리 → close() → 다시 accept() 순서로 반복합니다.
2. 구현이 단순하고 리소스 사용이 적습니다.
3. 한 클라이언트가 처리되는 동안 다른 클라이언트를 대기시킵니다.
4. 빠른 응답이 필요하지 않은 단순한 서비스에 사용합니다.

### ✅ 멀티프로세스 서버

1. accept() 후 fork()로 자식 프로세스를 생성하여 각 클라이언트를 처리합니다.
2. 부모는 accept() 반복, 자식은 클라이언트 처리 후 종료합니다.
3. 프로세스 간 격리로 안정성이 높고, 멀티코어 활용이 가능합니다.
4. 프로세스 생성 오버헤드가 크고, 메모리 사용량이 많습니다.
5. signal()로 SIGCHLD를 처리하거나 waitpid()를 사용합니다.

### ✅ 멀티스레드 서버

1. accept() 후 pthread_create()로 스레드를 생성하여 각 클라이언트를 처리합니다.
2. 메인 스레드는 accept() 반복, 워커 스레드는 클라이언트 처리합니다.
3. 프로세스보다 가벼우며, 메모리 공유로 효율적입니다.
4. 동기화 문제와 디버깅이 어렵습니다.
5. 미리 생성된 스레드 풀에서 스레드를 재사용하는 패턴입니다.

### ✅ 이벤트 기반 서버

1. select/epoll로 여러 소켓을 모니터링하고, 이벤트 발생 시 처리합니다.
2. 단일 스레드로 많은 연결을 처리할 수 있고, 리소스 효율이 좋습니다.
3. CPU 집약적 작업에 부적합하고, 블로킹 작업을 피해야 합니다.
4. 별도 스레드 풀에 작업을 위임하거나 비동기 I/O를 사용합니다.

### ✅ 패턴 비교

1. 멀티프로세스는 격리되지만 무겁고, 멀티스레드는 가볍지만 동기화가 필요합니다.
2. 멀티스레드는 스레드 생성, 이벤트 기반은 단일 스레드로 처리합니다.
3. 이벤트 기반이 대규모 연결에서 가장 효율적입니다.
4. 멀티프로세스가 가장 많고, 이벤트 기반이 가장 적습니다.

### ✅ 실무 적용

1. 반복 서버나 간단한 멀티스레드 서버가 적합합니다.
2. 이벤트 기반 서버(epoll)가 적합합니다.
3. 이벤트 기반 서버가 적합합니다.
4. 멀티스레드나 멀티프로세스가 적합합니다.

### ✅ 하이브리드 패턴

1. 이벤트 루프로 I/O를 처리하고, CPU 집약적 작업은 스레드 풀에 위임합니다.
2. I/O 효율성과 CPU 작업 처리를 모두 활용할 수 있습니다.
3. I/O와 CPU 작업이 혼재된 서버에서 사용합니다.

### ✅ 실무 / 설계 판단

1. 연결 수, 작업 특성, 하드웨어 리소스, 개발 복잡도를 고려합니다.
2. 이벤트 기반 서버가 가장 널리 사용됩니다.
3. 비동기 I/O와 이벤트 기반 아키텍처가 트렌드입니다.

### ✅ 요약 질문
