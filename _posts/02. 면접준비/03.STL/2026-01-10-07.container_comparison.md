## 📝 모범 답변

### ✅ 기본 개념

1. 시간 복잡도, 메모리 사용량, iterator 안정성, 캐시 친화성, 사용 패턴 등을 고려해야 합니다.
2. 컨테이너 선택에 따라 알고리즘의 시간 복잡도와 실제 실행 시간이 크게 달라질 수 있습니다.

### ✅ 시간 복잡도 비교

1. vector/deque: 앞뒤 O(1), 중간 O(N) / list/forward_list: 중간 O(1) / map/set: O(log N) / unordered_map/set: 평균 O(1)
2. map/set: O(log N) / unordered_map/set: 평균 O(1), 최악 O(N) / vector/deque: O(1) / list/forward_list: O(N)
3. vector/deque/array: O(1) / list/forward_list: O(N) / map/set: O(log N) / unordered_map/set: 평균 O(1)

### ✅ 메모리 사용량 비교

1. vector: 최소 오버헤드 / deque: 청크 관리 오버헤드 / list: 노드당 2포인터 / forward_list: 노드당 1포인터
2. map/set: 노드당 3포인터 + 색상 정보 / unordered_map/set: 버킷 배열 + 노드 오버헤드
3. stack/queue/priority_queue: 내부 컨테이너의 오버헤드만 발생합니다.

### ✅ 캐시 친화성 비교

1. vector, deque, array 등 연속 메모리 컨테이너가 캐시 친화성이 높습니다.
2. list, map, set 등 노드 기반 컨테이너가 캐시 친화성이 낮습니다.
3. 캐시 미스가 빈번하면 실제 실행 시간이 이론적 시간 복잡도보다 훨씬 느려질 수 있습니다.

### ✅ iterator 무효화 비교

1. vector, deque, unordered_map 등 재할당/rehash가 발생하는 컨테이너에서 자주 발생합니다.
2. list, forward_list, map, set 등 노드 기반 컨테이너에서 거의 발생하지 않습니다.
3. iterator 안정성이 중요하면 list, map, set 등을 선택해야 합니다.

### ✅ 실무 선택 기준

1. 중간 삽입/삭제가 많으면 list, 앞뒤 삽입이 많으면 deque, 뒤쪽만이면 vector를 선택합니다.
2. 정렬된 탐색이 필요하면 map/set, 빠른 평균 성능이 중요하면 unordered_map/set을 선택합니다.
3. 순서가 중요하면 vector, deque, list, map, set 등을 선택하고, 순서가 불필요하면 unordered_map/set을 선택합니다.
4. 메모리 최소화가 중요하면 vector, array를 선택하고, list나 map 계열은 피합니다.
5. 캐시 효율이 중요하면 vector, deque 등 연속 메모리 컨테이너를 선택합니다.

### ✅ 컨테이너 선택 의사결정 트리

1. 연속 메모리 필요 → vector/deque, 중간 삽입 빈번 → list, 정렬 필요 → map/set, 빠른 탐색 → unordered_map/set
2. 정렬된 순서 필요 → map/set, 빠른 평균 성능 → unordered_map/set, 중복 허용 → multiset/multimap
3. vector + unordered_map, vector + map, list + set 등이 자주 사용되는 조합입니다.
